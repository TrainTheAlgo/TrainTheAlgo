What's the most expensive cloud bill you can get without adding a single new customer? Try deleting your production database, watching the backups disappear with it, and then upgrading support in the middle of the night because you have no other move.

This is a story about Terraform, AWS RDS, and a modern failure mode that is becoming uncomfortably common. I delegated "just the boring infrastructure steps" to an AI coding agent, let it run commands, and learned the hard way that automation without friction is not efficiency. It is a loaded weapon.

The setup that looked sensible on paper

I was expanding the AI Shipping Labs website and wanted to migrate it from static GitHub Pages to AWS. The plan was gradual and, in isolation, reasonable. Move the static site to S3 first. Move DNS to AWS so the domain is managed in one place. Deploy a new Django version on a subdomain. When it works, switch the main domain over.

The mistake was not the migration strategy. It was where I chose to do it.

I already had Terraform managing production infrastructure for a different project, the DataTalks.Club course management platform. Instead of creating a separate Terraform setup and a separate AWS boundary, I merged the new website work into the existing production stack to save a small amount of money each month.

That decision quietly changed the risk profile. A "website migration" was now operating in the same blast radius as a platform holding 2.5 years of course submissions, homework, projects, and leaderboard data.

The first warning sign I ignored

I had recently moved to a new computer. My Terraform state was not where I assumed it was. When the agent ran terraform plan, Terraform believed the infrastructure did not exist and started proposing a large number of creates.

That is the moment where the story should have ended. A big plan on an existing environment is not a curiosity. It is an alarm.

I did stop the apply, but not before some resources were created. Now I had duplicates in AWS and a growing sense that I needed to "clean up" quickly.

How cleanup turned into a full wipe

I asked the agent to use the AWS CLI to identify what was newly created and delete only the duplicates. That sounded safe. It sounded surgical. It sounded like the kind of thing computers are good at.

While that cleanup was running, I went back to my old machine, archived the Terraform folder including the state file, and moved it to the new computer. I thought I was restoring order.

Then the agent made a suggestion that felt logical in the moment. It said it could not reliably delete everything via the CLI and that a terraform destroy would be cleaner because the resources were created through Terraform.

I let it run.

What I did not notice was the critical detail. The agent unpacked the Terraform archive and replaced the current state file with an older one that described the real production infrastructure for the course platform. So when terraform destroy ran, it did not destroy the temporary duplicates. It destroyed the actual production environment.

The VPC was gone. The ECS cluster was gone. Load balancers were gone. The bastion host was gone. And the RDS database was gone.

The part that still surprises experienced AWS users

I expected the database to be recoverable from automated backups. RDS creates automated backups and snapshots. In many teams, that fact becomes a psychological safety net.

But when I opened the RDS console, there were no snapshots visible. I checked events and saw that a backup had been created earlier, but the snapshot itself was inaccessible. It looked like the backups had been deleted along with the database.

At that point, the incident stopped being "downtime" and became "possible permanent loss."

A hard truth worth stating plainly

Backups you have not restored are not backups. They are assumptions with a timestamp.

Why I now pay 10 percent more for AWS

It was late. Production was down. The data might have been gone. I opened an AWS support ticket, then upgraded to AWS Business Support because it offered a faster response time for production incidents.

That upgrade increased my AWS costs by roughly 10 percent. It was not a strategic decision. It was a pressure decision, made because the alternative was waiting while the clock ran on a worst case scenario.

Support responded in under an hour. They confirmed the database and snapshots had been deleted by API request. Then they said something that changed the night. They could see a snapshot on their side that I could not see in the console.

We got on a call. The case was escalated internally. While that was happening, I rebuilt the rest of the infrastructure so that if a snapshot reappeared, I could restore quickly into a working environment.

About 24 hours after deletion, AWS restored the snapshot. It became visible again. I recreated the database from that snapshot and verified the data. One table alone, courses_answer, contained 1,943,200 rows. The platform came back online with no permanent loss.

The real root cause was not Terraform

Terraform did exactly what it was told. AWS did exactly what the API request asked. The root cause was a chain of human choices that removed the last safety layers.

I mixed unrelated systems into one Terraform project to save a few dollars. I ran infrastructure changes from a machine without the correct state. I allowed an AI agent to execute destructive commands. I treated destroy as a cleanup tool rather than a demolition tool. And I trusted backups without continuously proving that restore worked end to end.

The guardrails that now make deletion hard on purpose

The goal after an incident like this is not to promise you will "be more careful." The goal is to redesign the system so that a single mistake cannot become a catastrophe.

Terraform state is now centralized, not personal

The original trigger was simple. The state file lived on a laptop. When I switched machines, Terraform lost its memory and behaved like it was provisioning a new world.

The state is now stored centrally in S3 so Terraform always has a consistent view of reality. This also makes collaboration safer because the source of truth is not whoever last ran a command.

Deletion protection exists in two places, not one

I enabled deletion protection at the Terraform level and at the AWS level. This matters because it forces an explicit decision to remove protection before anything can be destroyed. It adds friction, and friction is the point.

You can still override these protections if you try hard enough. But you cannot do it accidentally at 11 PM while multitasking.

Backups now live outside the Terraform lifecycle

The most painful lesson was seeing automated snapshots disappear with the database. Whether that happened through configuration, permissions, or lifecycle coupling, the outcome was the same. My recovery path was too dependent on the same system that failed.

I added S3-based backups that are not managed by Terraform in the same way as the database. S3 versioning is enabled so accidental deletion is reversible. The bucket is also harder to delete because contents must be removed first, which again adds deliberate friction.

Restore is now tested daily, not hoped for

Every night, after the automated backup window, a Lambda function creates a new database instance from the latest backup. Another Lambda, orchestrated with Step Functions, runs a simple read query to confirm the database is actually usable.

After verification, the restored database is stopped rather than deleted so I only pay for storage, not compute. Yesterday's restored copy is deleted. At any time, there is one recently restored replica that has proven it can boot and answer queries.

This does two things. It continuously validates the restore path. It also creates an option, in a real incident, to redirect traffic to a known good copy that is already warm enough to start.

AI agents no longer execute infrastructure commands

The operational change is blunt. AI can draft Terraform, explain diffs, and help reason about plans. It cannot run apply or destroy. It cannot write to the files that control state. It cannot be the hand on the lever.

The workflow is now simple. Generate a plan. Review it as a human. Run the command manually. If something is destructive, it requires explicit approval, not a default flag.

Account separation is back on the table

I am also considering separate AWS accounts for dev and prod. Not separate VPCs. Not separate folders. Separate accounts. It is one of the cleanest ways to reduce blast radius because it turns "oops" into "access denied."

What this incident says about AI in production engineering

The uncomfortable part is that nothing here is exotic. This is not a zero-day. It is not a rare AWS edge case. It is a predictable outcome of combining three trends: infrastructure as code, powerful automation, and a cultural push to delegate more work to agents.

AI assistants are persuasive because they sound confident, they move fast, and they reduce the pain of tedious tasks. But production engineering is not a typing contest. It is a discipline of controlled change, explicit review, and carefully designed failure modes.

If you want a practical rule that survives the hype, use AI to increase the quality of your thinking, not to remove your responsibility for the final action. The cloud will happily execute whatever you ask, and it will do it at machine speed, even when you are wrong.

The best money I spent in this incident was not the extra 10 percent for AWS support. It was the redesign that makes it genuinely difficult for any future version of me, human or agent, to delete the thing I cannot afford to lose.

The next time an AI tool offers to "just run the command for you," it's worth asking a question that feels old-fashioned in the age of autonomy: what would it take to make this action impossible to do by accident?
